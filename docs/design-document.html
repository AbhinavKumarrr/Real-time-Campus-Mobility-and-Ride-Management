CampusRide — Design Document

Real-Time Campus Mobility & Ride Management Platform

Cult Open Projects 2027 · Software Development  |  Stack: MERN + Socket.IO

1. Problem Understanding

Large campuses such as IIT Roorkee depend on e-rickshaws for last-mile transport, but ride requests and driver availability are coordinated through fragmented, informal mechanisms. This causes inefficient resource use, delays, uneven demand distribution and a poor experience for both riders and drivers.

Goal: a centralized, real-time platform where passengers and drivers connect to discover, request, assign, track and complete rides reliably within a campus.

The engineering challenges mirrored from production ride-hailing systems are:

  • Real-time communication — both sides must see state changes instantly.
  • Consistent ride state — one well-defined lifecycle; a request assigned to exactly one driver.
  • Multi-user coordination — many passengers and drivers acting concurrently.
  • Geospatial handling — pickup, destination and driver positions on a map.
  • Insight — dashboards and demand analytics for drivers and operators.

Scope delivered: all seven mandatory features (authentication, availability, request workflow, real-time updates, lifecycle, driver dashboard, ratings) plus two bonus features — live map and demand analytics. Payments, scheduling and ML forecasting were deliberately scoped out to keep the core workflow reliable, following the guideline that "a reliable ride management system is preferable to numerous incomplete features."

Request vs. real-time split

  • REST is the source of truth for every state mutation (request, accept, start, complete, cancel, rate). Each endpoint validates, persists, then emits the result.
  • Socket.IO only pushes those changes to relevant clients — never the sole authority — so state stays consistent even when a socket reconnects.

4. Entity Relationship Diagram (ERD)

  • A passenger places many rides (1‑to‑many).
  • A driver fulfils many rides (a ride has 0‑or‑1 driver).
  • A ride receives 0‑or‑1 rating; a rating references its ride, passenger and driver.

Example — atomic accept: findOneAndUpdate({ _id, status:'requested', driver:null }, { driver, status:'accepted' }) → emits ride:assigned to the passenger and ride:requestClosed to other drivers.

6. Design Decisions

  1. REST for truth, sockets for push. All mutations go through validated REST endpoints; Socket.IO only broadcasts results — avoiding race conditions and surviving reconnects.
  2. Atomic single-driver assignment. Acceptance is a single conditional findOneAndUpdate (status:'requested', driver:null). MongoDB guarantees only one concurrent driver wins; others get a clear "already taken" response — satisfying "a ride can only be assigned to a single driver."
  3. Server-guarded state machine. Lifecycle transitions are validated server-side (allowed edges + participant/role checks), so the UI can never force an invalid state.
  4. Room-scoped events. Emitting to user:, ride:, drivers, passengers rooms delivers only relevant events — efficient and scalable.
  5. JWT + role middleware. Stateless auth scales horizontally; requireRole centralises authorisation; passwords are bcrypt-hashed and never serialized.
  6. Zero-setup dev experience. An in-memory MongoDB option with auto-seeding lets evaluators run the full stack with only Node installed; a real MONGO_URI is a one-line switch.
  7. Computed fare/distance. Haversine distance + a transparent base+per-km fare keep estimates deterministic without a paid routing API.
  8. Scalability path. Stateless API behind a load balancer; Socket.IO scales across nodes via a Redis adapter; indexes on hot query paths keep reads fast as data grows.

Trade-offs & future work

  • Matching is broadcast-to-all-online (simple, transparent) rather than proximity-ranked dispatch — a geo-nearest assignment is the natural next step.
  • Payments, scheduling and ML demand forecasting are scoped out, but the schema (timestamps, locations) already supports adding them.
